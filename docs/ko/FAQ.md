## 1. 라즈베리파이(32GB SD 카드)로 실제로 어디까지 가능한가요?

어떤 일에는 생각보다 멀리 가고, 어떤 일에는 턱없이 모자랍니다.
실제로 부딪히게 되는 순서대로 천장을 나열하면:

1. **RAM** (Pi 4는 2/4/8GB로 출시). 인터랙티브 Claude Code나 Codex 세션
   하나는 수백 MB를 쓰는 Node/Rust 프로세스라 4GB면 충분합니다. 파이를
   죽이는 건 *에이전트형* 부하입니다: 서브에이전트 fan-out, 큰 저장소
   인덱싱, 에이전트가 돌리는 `npm install`이나 테스트 스위트. 이런 작업은
   스왑까지 밀고 들어가는데, SD 카드 스왑은 기어가는 속도 **더하기** 카드
   수명 소모를 뜻합니다.
2. **SD 카드 I/O — 용량이 아니라.** 32GB면 *공간*은 넉넉합니다 (Raspberry
   Pi OS + 툴체인이 대략 9GB; 실제 설치 기준 사용률 33%). 문제는 랜덤 I/O
   속도와 쓰기 내구성입니다: 큰 저장소에서 `git status`와 패키지 설치가
   답답할 만큼 느리고, 로그 + 스왑이 저가 카드를 갉아먹습니다.
3. **CPU**는 맨 마지막. 터미널 작업과 채팅형 세션은 가볍습니다. 에이전트가
   촉발하는 빌드와 테스트 실행에서 작은 ARM 코어 네 개가 한계를 드러냅니다.

**파이가 기막히게 잘하는 것:**

- 항상 켜진 **tmux 호스트**: 세션을 걸어두고 지하철에서 휴대폰으로 붙기
  ([REMOTE-ACCESS.md §6.4](./REMOTE-ACCESS.md#64-워크플로우-d-항상-켜진-집-서버)).
- Tailscale **진입점 / 서브넷 라우터**: 노트북이 전부 가방 속에서 잠들어
  있어도 *항상* 닿는 머신.
- 가벼운 단일 에이전트 채팅형 세션; 벽시계 시간이 상관없는 밤샘 대기열 작업.

**파이가 못하는 것:** 멀티 에이전트 오케스트레이션, 큰 빌드, RAM을 많이
먹는 모든 것. 억지로 시키지 마세요 — 그 일은 맥 몫입니다.

**업그레이드, 가성비 순:**

1. 제대로 된 **5V/3A 전원 어댑터** — 전력이 모자란 파이는 느려지는 게
   아니라 *재부팅 루프*에 빠집니다
   ([§8.12](./REMOTE-ACCESS.md#812-라즈베리파이가-재부팅-루프--부하-걸리면-죽음)).
2. **USB SSD 부팅** — Pi 4에서 단일 변경으로 가장 큰 성능 개선; I/O 속도*와*
   내구성 걱정을 한 방에 해결합니다.
3. SD 카드 스왑 대신 **zram 스왑** (압축 RAM).
4. SD를 계속 쓴다면 고내구성 SD 카드.
5. 그래도 모자라면 → 중고 **미니 PC** (Intel N100, 8–16GB)가 스위트 스폿.
   파이는 다시 훌륭한 게이트웨이로 돌아갑니다.

늘 통하는 패턴: **파이는 현관, 맥은 근육.** 파이로의 SSH는 언제나 열려
있고, 무거운 세션은 맥에서 돕니다.

## 2. 중앙 호스트 없이 머신끼리 전부 동기화하면 안 되나요?

절반만 됩니다. 신경 쓰는 모든 걸 세 바구니로 나눠보세요:

| 바구니 | 예시 | 동기화? |
|--------|------|---------|
| **워크스페이스 안의 파일** | 코드, 생성된 리포트, diff, 문서 | ✓ 완벽하게 — git(또는 Syncthing)이 하라고 있는 일 |
| **계정 상태** | 세션, 채팅 기록, 트랜스크립트, 플러그인 캐시, 인증 | ✗ 일부러 안 함 — 머신 + 계정 로컬 |
| **살아있는 프로세스** | 실행 중인 에이전트, tmux 세션 | ✗ 물리적으로 불가능 — 실행 중인 프로세스는 파일이 아님 |

둘째 줄은 우연이 아닙니다. 계정 루트에는 머신 종속 경로, 캐시, 자격 증명이
들어 있습니다 (macOS에서 Claude 인증은 **키체인**에 살고, 키체인은 머신을
아예 떠날 수 없습니다 — [ARCHITECTURE.md §5](./ARCHITECTURE.md#5-알려진-한계)
참고). 계정 루트를 머신 간에 동기화하는 건, 이 라우터가 막으려고 존재하는
바로 그 상태 뒤섞임을 자초하는 일입니다.

그래서 순수 동기화 셋업의 상한선은 이렇습니다: **어디서나 같은 코드, 같은
산출물 — 하지만 기록과 실행 중인 세션은 머신마다 따로.** 원하는 게 "책상에서
일어나서 휴대폰으로 *같은 라이브 세션*을 이어받기"라면 — 그건 동기화 문제가
아니라 세션 호스트 문제이고, 답은 세션을 돌리는 머신 한 대에 나머지 전부가
붙는 구조입니다 (질문 3).

산출물이 호스트 사이를 더 자동으로 흐르게 만드는 건
[BACKLOG.md](../../BACKLOG.md)에서 추적 중입니다 ("cross-host
artifact/session continuity").

## 3. 모든 기기가 파이를 거쳐야 하나요?

아니요. Tailscale은 허브-스포크가 아니라 **메시(mesh)**입니다 — 모든 기기가
다른 모든 기기와 직접 통신합니다:

```
        ┌────────┐  direct   ┌─────────┐
        │ iPhone │◄─────────►│ M3 Max  │
        └───┬────┘           └────┬────┘
            │        direct       │
            │     ┌────────┐      │
            └────►│   Pi   │◄─────┘
                  └────────┘
     모든 간선이 직접 연결; 어느 것도 Pi를 거쳐 중계되지 않음
```

파이가 더해주는 건 라우팅이 아니라 **가용성**입니다: *항상* 켜져 있는
노드라서, 맥이 잠들었거나 가방 속에 있을 때 닿을 수 있는 곳이 파이인
겁니다. 맥이 깨어 있으면 휴대폰에서 `ssh doritos-m3-max`는 곧장 맥으로
갑니다.

파이의 용도: 장수 tmux 세션이 주차하는 곳, 절대 잠들지 않는 박스,
선택적으로 Tailscale 밖 LAN 기기를 위한 서브넷 라우터. 용도가 아닌 것:
의무 경유지.

## 4. 회사 망 피해서 집에서 회사 일 하면 왜 안 되나요?

기술적으로요? 됩니다. 이 저장소의 문서가 말 그대로 모든 부품 사용법을
가르쳐줍니다. 막는 건 처음부터 기술이 아니었으니, 솔직하게 쪼개봅니다 —
"집에서 회사 일"은 사실 세 가지 서로 다른 이야기이기 때문입니다:

| 케이스 | 모습 | 판정 |
|--------|------|------|
| **(a) 집에서 회사 노트북 + 회사 VPN** | 업무 머신을 집에 가져가 IT가 제공하는 방식으로 연결 | ✅ 그냥 재택근무입니다. 이렇게 하세요. |
| **(b) 승인된 채널을 통한 개인 기기** | BYOD 정책, 회사 VDI, 회사 Tailscale 테넌트, IT 서면 승인 | ✅ 물어보세요. 의외로 잘 나옵니다. |
| **(c) 회사 망을 *우회해서* 닿는 개인 인프라** | 업무 노트북의 무단 Tailscale, 회사 저장소를 집 머신에 복사, 모니터링을 피하는 터널 | ❌ 커리어가 끝나는 건 이쪽입니다. |

"**회사 망 피해서**"라는 표현이 정확히 (a)/(b)를 (c)로 바꾸는 지점입니다.
(c)가 정도의 차이가 아니라 종류부터 다른 이유:

- **정책 관점에서 이것은 유출(exfiltration)입니다.** 의도와 무관하게요.
  회사가 볼 수 없는 인프라 위로 회사 데이터가 흐르는 것 — *그게 정의
  그 자체*입니다.
- **다 보입니다.** 업무 노트북의 EDR/DLP는 VPN 클라이언트 설치를 보고,
  네트워크는 터널을 봅니다. 숨는 게 아니라 스스로 명단에 이름을 올리는
  겁니다.
- **당신의 처지가 바뀝니다.** *무엇이든* 유출되는 날 — 당신과 무관한
  것이어도 — 무단 데이터 채널을 가진 사람이 명단 맨 위에 오릅니다. "그냥
  야근하고 있었을 뿐인데요"는 사고 조사 앞에서 살아남지 못합니다.

좋은 소식이 있고, 진짜로 좋은 소식입니다: **회사에서 VPN 사용이 이미
공공연하다면, 서면 요청은 쉽습니다.** 암묵적 용인은 정책이 아니지만
([REMOTE-ACCESS.md §7.3](./REMOTE-ACCESS.md#73-회사-기기-허락-받으세요)),
답이 '예'일 거라는 강력한 증거*이긴* 합니다. '예'를 서면으로 받으면 케이스
(b)가 됩니다 — 편의는 그대로, 리스크는 없이.

그리고 애초에 아무 허락도 필요 없는 것도 기억하세요: 본인 머신과 본인
tailnet 위의 **개인** 프로젝트는 어디서든 자유입니다. 그 자유가 애초에
신원을 나눈 이유 전부입니다.

## 5. 어느 레이어가 무엇을 소유하나요? (머신 / 계정 / 도구 / 워크스페이스 / 브라우저)

멘털 모델: 세션은 **독립적인 축들의 교차점**이고, 축마다 소유하는 상태의
종류가 다릅니다. 라우터
([ARCHITECTURE.md §2](./ARCHITECTURE.md#2-핵심-아이디어-직교하는-축))는
각 축을 의도적으로 고르게 만들 뿐입니다.

```
Machine (예: M3 Max)       ← tmux 세션, 실행 중인 에이전트, Tailscale 노드
│                            신원, macOS 키체인: 절대 이 박스를 떠나지 않음
└── OS user (본인)
    ├── ACCOUNT 축 — 환경 변수로 선택      (인증, 청구, 기록)
    │     ~/.claude-personal   ~/.codex-personal
    │     ~/.claude-company    ~/.codex-company
    │
    ├── WORKSPACE 축 — cd로 선택           (파일, 로그, 프로젝트 설정)
    │     ~/dev/personal   → .ai-logs/  CLAUDE.md  .claude/  .mcp.json
    │     ~/dev/work       → .ai-logs/  CLAUDE.md  .claude/  .mcp.json
    │
    ├── BROWSER 축 — data-dir로 선택       (웹 로그인 쿠키)
    │     ~/.ai-browser-personal    ~/.ai-browser-company
    │     ~/.claude-app-personal    ~/.claude-app-company   (GUI 앱)
    │
    └── ~/.ai-shared — 유일하게 의도적으로 공유하는 곳
          (opt-in: 계정 간 공유하는 스킬 / 플러그인 마켓플레이스)
```

`ai claude company`에서 실제로 벌어지는 일:

```
                      ai claude company
                             │
             ┌───────────────┴───────────────┐
             ▼                               ▼
     ACCOUNT axis                     WORKSPACE axis
     CLAUDE_CONFIG_DIR=               cd ~/dev/work
     ~/.claude-company                       │
             └───────────────┬───────────────┘
                             ▼
                   claude process starts
                             │
       ┌─────────────────────┼─────────────────────┐
       ▼                     ▼                     ▼
 GLOBAL state          PROJECT state          transcript goes
 from the account      discovered walking     into the workspace:
 root: auth·settings   up from cwd:           ~/dev/work/.ai-logs/
 skills·plugins·MCP    CLAUDE.md · .claude/   claude/company-account/…
 registrations         .mcp.json
```

*(한글은 터미널에서 두 칸 폭이라 정렬이 깨지므로 레이블은 영어로 둡니다.
ACCOUNT axis = 계정 축 · WORKSPACE axis = 워크스페이스 축 · "claude process
starts" = claude 프로세스 시작 · GLOBAL state… = 계정 루트에서 읽는 전역
상태(인증·설정·스킬·플러그인·MCP 등록) · PROJECT state… = cwd에서 위로
올라가며 발견하는 프로젝트 상태 · transcript… = 트랜스크립트는
워크스페이스의 `~/dev/work/.ai-logs/claude/company-account/…`에 기록.)*

"인식은 대체 어디서 시작되나?"의 답이 저 세 번째 화살표 쌍입니다: **전역
상태는 환경 변수를 따라가고, 프로젝트 상태는 지금 서 있는 디렉터리를
따라갑니다.** 워크스페이스가 "가상"인 건 그저 평범한 디렉터리라는
의미에서일 뿐입니다 — 도구들은 실제로 `cwd`에서 위로 올라가며 프로젝트
설정을 해석하므로, 디렉터리 경계가 *곧* 워크스페이스 경계입니다. 라우터의
일이란 결국, 도구가 시작되기 전에 환경 변수와 cwd가 같은 신원을 가리키게
만드는 것 전부입니다.

## 6. CLI, 웹, GUI는 무엇을 공유하나요? 스킬, MCP, 플러그인은요?

공유 매트릭스 전체 — "계정" 열은 같은 머신을 전제하고, 마지막 열은 그것이
다른 머신까지 따라오는지를 묻습니다:

| 상태 | 위치 | 계정 간 공유? | 머신 간 이동? |
|------|------|---------------|---------------|
| Claude CLI 인증 | macOS 키체인 (설정 디렉터리별 키) | ✗ | ✗ 키체인은 머신 종속 |
| Codex CLI 인증 | `$CODEX_HOME/auth.json` (기본은 파일 모드; 키링은 선택) | ✗ | ✗ (파일이긴 하나 자격 증명 복사는 금물) |
| CLI 채팅 기록 / 세션 | 계정 루트 | ✗ | ✗ |
| 스킬, 플러그인 | 계정 루트 — 또는 `~/.ai-shared` (opt-in 공유 저장소) ⁽¹⁾ | opt-in ✓ | ✗ (하지만 git 관리 가능) |
| MCP 서버, 전역 | 계정 루트 설정 (계정별 토큰) | ✗ | ✗ |
| MCP 서버, 프로젝트 | 워크스페이스의 `.mcp.json` | 계정이 아니라 *워크스페이스*를 따라감 | ✓ 저장소가 동기화되면 |
| 프로젝트 메모리 (`CLAUDE.md`, `.claude/`) | 워크스페이스 | 워크스페이스를 따라감 | ✓ git으로 |
| 터미널 트랜스크립트 (`.ai-logs/`) | 워크스페이스, **`ai`를 실행한 호스트에** | 워크스페이스를 따라감 | ✗ 동기화하지 않는 한 |
| 웹 채팅 (claude.ai / chatgpt.com) | **프로바이더 서버** | ✗ 웹 로그인 별도 | ✓ 어느 기기든, 같은 로그인 |
| GUI 앱 데이터 (Claude.app) | `~/.claude-app-<id>` | ✗ | ✗ |
| tmux 세션, 실행 중인 에이전트 | 그 머신 | 해당 없음 | ✗ — 세션 호스트가 존재하는 이유 |

세 줄은 따로 강조할 만합니다:

- **웹만 혼자 다릅니다.** CLI 기록은 로컬(계정 루트, 이 머신)이지만, *웹*
  채팅 기록은 Anthropic/OpenAI 서버에 살면서 웹 로그인을 따라 어느
  기기로든 갑니다. 같은 "계정" 개념이 표면에 따라 완전히 다른 곳에
  착지하는 겁니다. 브라우저 축이 존재하는 이유가 이것입니다:
  `~/.claude-company`가 회사 CLI 인증을 격리하는 것과 똑같은 방식으로,
  `~/.ai-browser-company`는 *회사 웹 로그인의 쿠키*를 격리합니다.
- **MCP는 스코프에 따라 갈립니다.** 프로젝트 스코프 `.mcp.json`은 저장소와
  함께 여행합니다 — 어떤 계정이든, 그 워크스페이스를 여는 어떤 머신에서든
  보입니다. 전역 등록된 MCP 서버(토큰 포함)는 계정 루트에 앉아서 거기
  머뭅니다. MCP를 손으로 배선할 때는
  ([ARCHITECTURE.md §6](./ARCHITECTURE.md#6-mcp-v1에서는-일부러-범위-밖))
  공유 가능하고 시크릿 없는 서버는 `.mcp.json`에, 토큰이 붙는 건 전부 계정
  루트에 두세요.
- **스킬/플러그인은 기본이 계정별**이라, 같은 스킬을 두 번(계정마다)
  설치하게 됩니다. `~/.ai-shared` 심링크 패턴은 인증/세션은 갈라둔 채
  중복만 접어주는데, 이걸 `ai` 명령으로 굳히는 건
  [BACKLOG.md](../../BACKLOG.md) 후보입니다 ("shared-store command").

같은 매트릭스를 그림으로 보면:

```
┌─ this machine ────────────────────────────────────────────────────┐
│                                                                   │
│   personal                             company                    │
│   ~/.claude-personal                   ~/.claude-company          │
│   ~/.codex-personal                    ~/.codex-company           │
│   (auth · history · plugins)           (auth · history · plugins) │
│         │                                     │                   │
│         │ opt-in                       opt-in │                   │
│         └────────►  ~/.ai-shared  ◄───────────┘                   │
│                     skills · marketplaces                         │
│                     (account-neutral)                             │
│                                                                   │
│   ~/.ai-browser-personal               ~/.ai-browser-company      │
│   (web cookies) ····································┐             │
│         :                                     :     :             │
│   ~/dev/personal                       ~/dev/work   :             │
│   code · CLAUDE.md ·                   code · CLAUDE.md ·         │
│   .mcp.json · .ai-logs                 .mcp.json · .ai-logs       │
│         │                                     │     :             │
└─────────┼─────────────────────────────────────┼─────┼─────────────┘
          ▼                                     ▼     ▼
     git remote                            git remote  claude.ai / chatgpt
     (workspace state                                  (server-side history,
      travels here)                                     follows the web login)
```

*(레이블은 정렬 보존을 위해 영어로 둡니다. this machine = 이 머신 · auth ·
history · plugins = 인증·기록·플러그인 · skills · marketplaces
(account-neutral) = 스킬·마켓플레이스(opt-in, 계정 중립) · web cookies = 웹
쿠키 · "workspace state travels here" = 워크스페이스 상태는 여기로 이동 ·
"server-side history, follows the web login" = 서버 측 기록, 웹 로그인을
따라감.)*

머신 박스 안의 모든 것은, 선이 박스 밖으로 나가지 않는 한 머신에
남습니다: 워크스페이스는 git으로 이동하고, 웹 기록은 프로바이더 곁에
살고, 공유 저장소는 계정 사이에 놓인 유일한 의도적 다리입니다.
